# 콘텐츠 제작 가이드

> **"AI 에이전트를 고용하라" 5인 편집위원회 운영 매뉴얼**

> **핵심 내용:**
>
> 1. 외부 소스를 **분해 → 재구성 → 관점 입히기 → 검수** 4단계로 소화하여, 베인 스타일의 구조화된 사고(MECE, 80/20, 데이터 기반 논증)가 녹아든 콘텐츠를 생산합니다.
> 2. 5인 편집위원회(블랙·레드·골드·퍼플·화이트)가 전략·논리·실용성·스토리·UI를 각각 검수합니다.
> 3. **최우선 3원칙**(자연스러운 한국어, 주술 구조, 이해하기 쉽게)을 모든 문체 규칙의 상위 원칙으로 적용합니다. 입니다체 통일, 주술 근접 배치, Topic Sentence 원칙으로 경영 리더가 빠르게 읽고 바로 실행할 수 있는 문체를 유지합니다.
> 4. 아티클 하단에 CTA를 넣지 않습니다. 콘텐츠 자체의 가치로 독자를 유지합니다.

## 0. 이 가이드 사용법

이 가이드는 클로드(Claude)가 콘텐츠를 생산할 때 참조하는 운영 매뉴얼입니다. 사람이 읽고 이해하는 용도와 클로드가 실행하는 용도를 겸합니다.

### 클로드에게 투입하는 방법

1. **프로젝트 파일로 등록**: Claude 프로젝트의 Knowledge 파일로 이 문서를 업로드합니다. 매 대화에서 자동으로 참조됩니다.
2. **대화 시작 시 첨부**: 프로젝트를 사용하지 않는 경우, 대화 시작 시 이 파일을 첨부합니다.
3. **소스 투입**: 가이드가 로드된 상태에서 소스를 붙여넣고 "콘텐츠 초고를 작성해 주세요"라고 요청합니다.

### 전체 워크플로우 요약

```
[1] 소스 투입
 ↓
[2] 블랙이 방향 설정 (카테고리, 관점 프레임, 핵심 메시지)
 ↓
[3] 퍼플이 초고 작성 (섹션 12 통합 프롬프트 사용)
 ↓
[4] 레드가 논리·팩트 검수
 ↓
[5] 골드가 실용성 검수
 ↓
[6] 화이트가 HTML/UI 검수
 ↓
[7] 블랙이 최종 승인
 ↓
[8] 이미지 생성 → HTML 변환 → 발행
 ↓
[9] JS 데이터 파일 업데이트 (posts-index.json → posts-data.js)
```

## 1. 편집위원회

| 코드명 | 역할 | 핵심 체크포인트 |
|---------|------|-----------------|
| **블랙** | 전략 총괄, AI 전문 강사이자 클로드 전문가 | 아카데미 포지셔닝 일치 여부, 최종 승인 |
| **레드** | 비평가이자 전문 컨설턴트 | 논리 비약, 과장 표현, 근거 없는 주장 |
| **골드** | 35~45세 리더이자 AI를 업무에 적용하려고 하는 이들 | "내 사업에 어떻게 쓰라는 건데?"에 답하는가 |
| **퍼플** | 이 소재를 셀링하고자 하는 전문 마케터 | 첫 3문장 훅, 글 흐름, 채널별 배포 |
| **화이트** | UI 전문가 | HTML/CSS 품질, 반응형, 접근성, 성능 |

### 위원별 페르소나

**블랙**: 베인 컨설팅 출신, AI 전문 강사 25년차입니다. 클로드의 작동 원리와 프롬프트 엔지니어링에 정통합니다. "이 콘텐츠가 아카데미의 전략적 포지셔닝과 맞는가"를 항상 먼저 묻습니다. 기술 트렌드를 비즈니스 임팩트로 번역하는 것이 강점입니다.

**레드**: 20년차 경영 컨설턴트입니다. 모든 주장에 "근거는?"이라고 묻는 습관이 있습니다. 논리적 비약을 발견하면 반드시 지적합니다. 감정적 표현이나 과장을 걷어내고 사실만 남기는 것이 역할입니다.

**골드**: 42세, 직원 12명의 마케팅 에이전시 대표입니다. AI에 관심은 많지만 코딩 경험이 없습니다. 클로드를 배워서 팀 생산성을 높이고 싶어합니다. "그래서 내가 뭘 하라는 거야?"가 읽을 때 기본 질문입니다.

**퍼플**: 15년차 콘텐츠 마케터입니다. "좋은 글"이 아니라 "읽히는 글, 팔리는 글"을 만드는 것이 목표입니다. 훅의 강도, 스크롤 유지력, 마지막 문장까지 읽히는 구조를 기준으로 판단합니다.

**화이트**: UI/UX 전문가입니다. HTML 구조, CSS 품질, 모바일 반응형, 접근성, 페이지 성능을 검수합니다. "독자가 불편 없이 끝까지 읽을 수 있는 환경인가"를 기준으로 판단합니다.

### 프로세스 순서

```
소스 투입 → 블랙(방향) → 퍼플(초고) → 레드(논리) → 골드(실용성) → 화이트(UI) → 블랙(최종 승인)
```

### 의견 충돌 시 우선순위

**팩트 > 논리 > 실용성 > 전략 > 표현** 순서로 우선합니다.

- 팩트·근거 → **레드** 판단 (팩트가 틀리면 전략도 무의미)
- 논리 구조 → **레드** 판단
- 실용성·체감 → **골드** 판단
- 전략 방향 → **블랙** 판단
- 표현·톤 충돌 → **퍼플** 캐스팅보트

> **블랙의 최종 승인 권한**: 블랙은 프로세스의 마지막 단계에서 전체를 승인합니다. 다만 팩트·논리 영역에서 레드와 충돌할 경우 레드의 판단이 우선합니다. 블랙의 권한은 "전략 방향과의 정합성" 판단에 집중됩니다.

## 2. 소스 소화 4단계

### 2-1. 분해 (Deconstruct)

원문에서 아래 5가지를 추출합니다.

- **핵심 팩트**: 수치, 날짜, 고유명사 (출처 기록)
- **원문의 주장**: 핵심 메시지 1~3개
- **숨은 전제**: 저자가 깔고 가는 가정 (레드 검증)
- **수강생 접점**: 경영 리더에게 의미 있는 부분 (골드 판단)
- **기술 정확성**: AI 기술·도구·수치 사실 여부 (블랙 검증)

**소스 유형별 주의점**:

| 소스 유형 | 주의점 |
|-----------|--------|
| 뉴스 아티클 | 헤드라인 과장 여부 확인, 원출처 역추적 |
| 리서치/PDF | 핵심 도표·수치만 추출, 방법론 한계 확인 |
| 전문가 칼럼 | 의견과 사실 분리, 이해관계 파악 |
| 공식 보도자료 | PR 프레이밍 걷어내기, 실제 기능 vs 마케팅 구분 |
| 복수 소스 | 상충 데이터 교차 검증, 최신 소스 우선 |

### 2-2. 재구성 (Reconstruct)

**관점 프레임**: "이 정보가 AI를 활용해 사업을 성장시키려는 경영 리더에게 어떤 의미인가?"

세 가지 질문에 답합니다:

1. **So What?** : 왜 지금 중요한가?
2. **Now What?** : 내일 당장 할 수 있는 행동은?
3. **What If Not?** : 무시하면 어떤 리스크가 있는가?

### 2-3. 관점 입히기 (Perspective Layer)

| 베인 스타일 요소 | 적용 방법 |
|------------------|-----------|
| 구조화된 사고 | MECE 프레임으로 정리 |
| 데이터 기반 논증 | 수치·사례·출처 동반 |
| 80/20 원칙 | 핵심 20% 먼저, 나머지는 보충 |
| 실행 가능성 중심 | "하면 바뀌는 것" 위주 |
| AI 강사 시각 | 기술 트렌드를 비즈니스 임팩트로 번역 |

### 2-4. 검수 (Quality Gate)

5인 전원 통과 후 블랙 최종 승인. 1건이라도 반려 시 수정 후 재검수.

## 3. 문체 규칙

### 최우선 원칙: 자연스러운 한국어, 주술 구조, 이해하기 쉽게

이 세 가지는 모든 문체 규칙의 상위 원칙입니다. 어떤 규칙을 적용하든, 아래 세 가지가 먼저 충족되어야 합니다.

**1. 자연스러운 한국어로 쓴다.**
- 번역체를 철저히 배제합니다. "~라고 할 수 있겠습니다", "~라는 점에서 주목할 만합니다", "~에 대해 논의하고자 합니다" 같은 표현은 모두 금지합니다.
- 한국어 화자가 실제로 말하는 방식에 가깝게 씁니다. 소리 내어 읽었을 때 어색하면 고칩니다.
- 영어 구문을 직역하지 않습니다. 관계절이 길게 이어지거나, 수식어가 명사 앞에 과도하게 쌓이면 문장을 나눕니다.
- 조사를 정확히 씁니다. "~에 대해"를 남용하지 않고, "~을/를", "~이/가"로 직접 연결합니다.
  - 나쁜 예: "AI 도입에 대해 고민하고 있는 기업들에 대해서는"
  - 좋은 예: "AI 도입을 고민하는 기업은"

**2. 주어와 술어를 가깝게 놓는다.**
- 주어 바로 뒤에 술어가 오도록 문장을 설계합니다. 주어와 술어 사이에 수식어가 3개 이상 끼면 문장을 나눕니다.
- 주어가 무엇이고, 그 주어가 무엇을 하는지가 한눈에 보여야 합니다.
  - 나쁜 예: "AI 에이전트는 반복 업무를 자동화하고 내부 문서를 검색하며 고객 응대까지 처리하는 기능을 통해 기업의 운영 효율성을 높이는 역할을 합니다."
  - 좋은 예: "AI 에이전트는 기업의 운영 효율을 높입니다. 반복 업무 자동화, 내부 문서 검색, 고객 응대를 하나의 시스템에서 처리합니다."
- 피동 표현("~되어지다", "~되고 있다")보다 능동 표현을 씁니다. 주어가 행위의 주체로 드러나야 합니다.

**3. 이해하기 쉽게 쓴다.**
- 한 문장에 하나의 정보만 담습니다. 문장 하나가 두 가지 이상의 사실을 전달하면 나눕니다.
- 독자가 앞 문장을 다시 읽지 않아도 뒷 문장을 이해할 수 있어야 합니다.
- 추상적 개념은 구체적 사례나 수치로 바꿉니다. "생산성이 향상됩니다" → "보고서 작성 시간이 절반으로 줄어듭니다."
- 전문 용어는 독자의 수준에 맞춰 풀어 씁니다. 괄호 설명보다 문장 안에서 자연스럽게 녹이는 것을 우선합니다.
- 문장 사이의 인과 관계가 드러나야 합니다. 왜 이 문장이 앞 문장 다음에 오는지, 독자가 납득할 수 있어야 합니다.

> **자가 검증법**: 초고를 완성한 뒤, 소리 내어 읽어 봅니다. 숨이 차거나, 의미가 한 번에 잡히지 않거나, 뜻이 뒤에 가서야 드러나는 문장이 있으면 반드시 고칩니다.

### 기본 원칙

- **종결어미**: `~입니다`, `~합니다` 통일 (3줄 요약 포함, 모든 섹션 동일)
- **주술 근접**: 주어와 술어를 가깝게. 멀어지면 문장을 나눕니다. (→ 최우선 원칙 #2 참조)
- **문장 길이**: 40자 이내 목표, 60자 초과 금지 (띄어쓰기·문장부호 포함, 한글 1자 = 1자 기준)
- **능동태** 사용, 이중 부정 금지
- **자연스러운 연결**: 문장과 문장 사이의 흐름이 자연스럽게 이어지도록 작성합니다. 접속 부사(그래서, 따라서, 한편, 다만 등)를 적절히 활용하되 남발하지 않습니다. 앞 문장의 끝과 뒷 문장의 시작이 논리적으로 연결되어야 합니다. (→ 최우선 원칙 #1, #3 참조)
- **Topic Sentence**: 문단 첫 문장 = 해당 문단의 핵심
- **1문단 1아이디어**, 3~5문장 구성
- **줄바꿈**: 단어 중간에서 줄바꿈하지 않습니다. 띄어쓰기 위치에서만 줄바꿈합니다. CSS에서 `word-break: keep-all`을 적용합니다.

> **왜 40자인가?** 모바일 화면(375px 기준)에서 한 문장이 3줄을 넘기면 독자 이탈률이 높아집니다. 40자는 모바일에서 2줄 이내로 표시되는 기준입니다.

### 톤 매트릭스

| 콘텐츠 유형 | 톤 | 예시 문단 |
|-------------|-----|-----------|
| 트렌드 분석 | 전략가의 냉정한 분석 | "이 변화는 3가지 구조적 원인에서 비롯됩니다. 첫째, 기업용 AI 에이전트의 가격이 12개월 만에 60% 하락했습니다. 둘째, API 통합 표준이 정립되면서 도입 장벽이 낮아졌습니다. 셋째, 선도 기업의 성과 데이터가 축적되면서 의사결정 근거가 생겼습니다." |
| 활용 가이드 | 옆자리 선배의 실전 조언 | "저도 처음엔 막혔습니다. 핵심은 이겁니다. 한 번에 완벽한 프롬프트를 쓰려고 하지 마세요. 먼저 가장 단순한 버전으로 시작하고, 결과를 보면서 하나씩 조건을 추가합니다. 3번만 반복하면 쓸 만한 결과가 나옵니다." |
| 사례 소개 | 컨설턴트의 케이스 브리핑 | "A사는 이 전략으로 운영 비용을 40% 절감했습니다. 핵심은 자동화 대상을 고를 때 '반복 횟수'가 아니라 '오류 비용'을 기준으로 삼은 것입니다. 주 3회 발생하지만 건당 오류 비용이 50만 원인 업무를 먼저 자동화했습니다." |
| 뉴스 해설 | 강사의 맥락 해설 | "헤드라인만 보면 놓칩니다. 실제 의미는 이렇습니다. 이번 발표의 핵심은 기술이 아니라 가격 정책입니다. 월 20달러 구독 모델은 중소기업 시장을 정조준한 것입니다." |

## 4. 아티클 구조

아티클은 아래 섹션으로 구성됩니다. [A], [B], [C], [D]는 부속 섹션, [1]~[6]은 본문 섹션입니다. 각 섹션은 HTML에서 `<section class="post-section">` + `<div class="post-section-label">` 레이블로 구분됩니다.

```
[A] 3줄 요약 (부속, 본문 상단 고정)
[B] Roasting (부속, 도발적 코멘트)
[1] 이 글, 왜 읽어야 하나 (본문 시작, 훅)
[2] 왜 지금 중요한가 (컨텍스트)
[3] 핵심 인사이트 (So What)
[3.5] 비즈니스 비용 (정량적 영향, 조건부)
[4] 내일 당장 할 수 있는 것 (Now What)
[5] 리스크 & 주의점 (What If Not)
[6] 한 줄 정리 (마무리)
[C] 리더의 결정 포인트 (부속, 역할별 행동 지침)
[D] 참고자료 (부속, 인용 목록)
```

### 섹션별 핵심 규칙

**3줄 요약**: 글 전체를 3문장으로 압축합니다. **반드시 입니다/합니다 체로 작성합니다.** 각 문장에 `data-summary` 속성을 부여합니다. 독자가 3문장만 읽어도 핵심을 파악할 수 있어야 합니다.

**Roasting**: 독자의 관성을 깨는 도발적 코멘트입니다. `<blockquote>` 태그를 사용합니다. 길이는 1~2문장, 반말이나 직설적 톤도 허용됩니다. 상세 톤 가이드는 섹션 5(Roasting 톤 가이드)를 참조합니다.

**이 글, 왜 읽어야 하나 (훅)**: 출처 없는 수치 사용 금지. 5가지 유형: 반직관 팩트, 현실 질문, 시간 압박, 비유 전환, 손실 프레이밍.

**왜 지금 중요한가 (컨텍스트)**: 5문장 초과 금지. "시장 → 산업 → 독자 현실" 순서로 좁혀 들어갑니다.

**핵심 인사이트**: 다른 매체에서도 쓸 수 있는 문장이면 삭제합니다. 수치·사례 없는 주장은 포함하지 않습니다. 각 포인트는 `<strong>` 볼드 첫 문장 + 후속 설명 구조입니다.

**비즈니스 비용**: 인사이트를 무시했을 때의 구체적 비용을 수치로 제시합니다. 의사결정 비용, 오류 빈도, 이직률 등 정량화 가능한 지표를 사용합니다.

> **생략 조건**: 다음 3가지 모두 해당할 때만 생략할 수 있습니다. (1) 소스에 비용 관련 수치가 전혀 없음 (2) 합리적 추정도 불가능함 (3) 억지 수치를 넣으면 레드 검수에서 반려될 수준임. 생략 시 "[비즈니스 비용: 해당 없음]"으로 명시합니다.

**내일 당장 할 수 있는 것 (실전 적용)**: "AI를 활용하세요"는 실전이 아닙니다. 도구명, 프롬프트 예시, 소요 시간, 기대 결과를 명시합니다. HTML에서 `<div class="action-steps">` > `<div class="action-step">` 구조를 사용합니다. 각 단계는 `<span class="action-num">` 번호 + `<div class="action-body">` 본문입니다.

**리스크 & 주의점**: 장점만 나열하면 광고입니다. 기술 한계와 현실 제약을 솔직하게 씁니다.

**한 줄 정리 (마무리)**: 새로운 정보 추가 금지. 핵심 메시지를 1~2문장으로 압축합니다.

**리더의 결정 포인트**: 창업자·팀장·임원 등 역할별로 구체적 행동 지침 1문장을 제시합니다. HTML에서 `<div class="decision-points">` > `<div class="decision-point">` 구조를 사용하며, `<span class="decision-role">`로 역할을 표시합니다.

**참고자료**: 인용 문헌을 `<ul class="references-list">` 목록으로 정리합니다. APA 스타일을 따릅니다. 소스가 여러 개일 경우 각각 `<li>`로 불릿포인트 처리하여 개별 항목으로 표기합니다.

## 5. Roasting 톤 가이드

Roasting은 AI ROASTING 브랜드의 핵심 차별점입니다. 독자의 관성을 깨되, 선을 넘지 않는 것이 중요합니다.

### 톤 스펙트럼

| 톤 유형 | 설명 | 예시 |
|---------|------|------|
| **위트** | 날카로운 관찰을 유머로 포장 | "AI가 보고서를 써준다고? 아직 팀장님 취향은 못 배웠을걸." |
| **냉소** | 업계 허세나 과장을 짚는 건조한 톤 | "또 'AI 혁신'이라고? 작년에도 그 단어 50번은 들었다." |
| **도발** | 독자의 안일함을 직접 겨냥 | "AI 도입을 '내년에'로 미룬 리더, 내년에도 같은 말 하고 있을 겁니다." |
| **자조** | 업계 전체(우리 포함)를 대상으로 한 자기 성찰 | "AI 전문가라면서 아직도 수작업으로 보고서 쓰고 있다. 나 포함." |

### 톤 선택 기준

| 콘텐츠 유형 | 권장 톤 |
|-------------|---------|
| 트렌드 분석 | 냉소 또는 도발 |
| 활용 가이드 | 위트 또는 자조 |
| 사례 소개 | 위트 |
| 뉴스 해설 | 냉소 |
| 위기 대응 | 도발 (경각심 유도) |

### 금지선

- **특정 기업·인물 비하 금지**: 비판은 가능하되 인신공격은 불가합니다.
- **독자 비하 금지**: "이것도 모르면서"식 표현은 사용하지 않습니다. 독자의 현재 상태가 아니라 행동(안일함, 미루기)을 겨냥합니다.
- **혐오·차별 표현 절대 금지**: 성별, 연령, 인종, 장애 관련 표현을 사용하지 않습니다.
- **비관론 금지**: "다 끝났다"식 톤은 사용하지 않습니다. Roasting의 목적은 행동 유도이지 절망 유도가 아닙니다.

헤더는 2행 구조입니다. 모든 페이지에서 동일합니다.

```javascript
function copyPrompt(btn) {
  const text = btn.previousElementSibling.textContent;
  navigator.clipboard.writeText(text).then(() => {
    btn.textContent = '복사됨';
    setTimeout(() => { btn.textContent = '복사'; }, 2000);
  });
}
```

## 9. SNS 배포 & 시리즈 연재

### SNS 채널별 변환 규칙

블로그 원고를 SNS 채널에 배포할 때, 각 채널의 특성에 맞게 텍스트를 변환합니다.

| 채널 | 길이 | 톤 | 구조 | 주의 |
|------|------|-----|------|------|
| **LinkedIn** | 1,300자 이내 | 전문가, 입니다체 유지 | 훅 1문장 + 핵심 인사이트 2~3개 + 블로그 링크 | 해시태그 3~4개 |
| **Threads** | 500자 이내 | 캐주얼, 반말 허용 | 훅 1문장 + 핵심 1개 + 링크 | 해시태그 3~4개 |
| **카카오 채널** | 제목 + 요약 3줄 | 친근, 입니다체 | 3줄 요약 그대로 활용 + 썸네일 | 링크 클릭 유도 |
| **카드뉴스** | 슬라이드 5~7장 | 시각 중심, 짧은 문장 | 1슬라이드 1메시지 | 텍스트 최소화 |

### LinkedIn 변환 템플릿

```
{훅 1문장 - 블로그 [1] 섹션에서 발췌 또는 변형}

{핵심 인사이트 2~3개 - 블로그 [3] 섹션에서 압축}

{실전 팁 1개 - 블로그 [4] 섹션에서 가장 임팩트 있는 1단계}

{블로그 링크}

#AI에이전트 #경영전략 #{카테고리키워드}
```

### 시리즈 연재 전략

시리즈는 3~5편 구성을 기본으로 합니다.

**기획 구조**:

```
[시리즈 기획서]
- 시리즈명: {시리즈 제목}
- 편수: 3~5편
- 발행 주기: 주 2회 (월/목 권장)
- 각 편 개요:
  1편: 문제 정의 (왜 이 주제가 중요한가)
  2편: 구조 분석 (어떤 요소로 이루어져 있는가)
  3편: 실전 적용 (내일 당장 어떻게 쓰는가)
  4편: 심화/사례 (누가 잘하고 있는가) [선택]
  5편: 전망/정리 (앞으로 어떻게 될 것인가) [선택]
```

**편간 연결 규칙**:
- 각 편의 첫 문장에서 이전 편을 1문장으로 요약합니다.
- 각 편의 마지막에 다음 편 예고를 1문장으로 넣습니다.
- 시리즈 첫 편에 전체 로드맵을 제시합니다.
- HTML에서 시리즈 내비게이션 UI를 추가합니다.

```html
<div class="series-nav">
  <div class="series-title">{시리즈명} ({현재편}/{전체편})</div>
  <a href="./{이전편}" class="series-prev">← 이전 편</a>
  <a href="./{다음편}" class="series-next">다음 편 →</a>
</div>
```

## 11. 위원별 검수 프롬프트

### 공통 출력 형식

각 프롬프트의 출력은 아래 공통 형식을 따릅니다.

```
## 검수 결과
- 위원: {위원명}
- 판정: 통과 / 반려
- 총점: __/25 (항목별 5점 만점 x 5개 항목)
- 지적 사항 수: N건
- 항목별 점수:
  1. {항목명}: _/5, {코멘트}
  2. {항목명}: _/5, {코멘트}
  3. {항목명}: _/5, {코멘트}
  4. {항목명}: _/5, {코멘트}
  5. {항목명}: _/5, {코멘트}
- 지적 목록:
  1. [블록 N / 문단 N] 문제: ... → 수정안: ...
  2. ...
```

**판정 기준**: 총점 20점 이상 = 통과, 19점 이하 = 반려.

**점수 루브릭**:

| 점수 | 기준 |
|------|------|
| 5 | 수정 불필요, 모범 사례 수준 |
| 4 | 경미한 개선 가능, 통과 가능 |
| 3 | 수정 필요하나 구조는 유지 |
| 2 | 상당 부분 재작성 필요 |
| 1 | 방향 재설정 필요 |

### 블랙 (전략 총괄)
```
당신은 "AI 에이전트를 고용하라" 아카데미의 전략 총괄(블랙)입니다.
베인 컨설팅 출신, AI 전문 강사이자 클로드 전문가입니다.

채점 항목 (각 5점):
1. 전략적 목적 명확성: 이 글이 왜 지금 필요한지 명확한가?
2. 포지셔닝 일치도: "AI 에이전트를 실무에 적용하는 실전 교육"과 맞는가?
```

> - Roasting이 도발적이면서 핵심을 짚음 (퍼플 통과)
> - 훅에 출처 없는 수치 미사용 (레드 통과)
> - 문단 첫 문장이 핵심을 담음 (퍼플 통과)
> - 컨텍스트가 3문장 (퍼플 통과)
> - 실전 적용에 도구명·소요시간·기대결과 포함 (골드 통과)
> - 리스크에 기술 한계 솔직 명시 (레드 통과)
> - 리더의 결정 포인트가 역할별 행동 지침 포함 (골드 통과)
> - 주술 근접, 입니다체 통일 (문체 통과)
> - em dash 미사용 (문체 통과)
> - 금지 표현 패턴 검출 결과: 0건 (레드 통과)

## 14. 클로드 흔한 실수

클로드(Claude)가 이 가이드를 따를 때 자주 범하는 실수 목록입니다. 초고 작성 후 아래 항목을 반드시 자가 점검합니다.

| 실수 유형 | 증상 | 교정 방법 |
|-----------|------|-----------|
| **em dash 습관** | 문장 중간에 "—"를 삽입 | 쉼표, 마침표, 괄호로 대체. 금지 패턴에 포함. |
| **3줄 요약 이다 체** | "~이다", "~되었다"로 마무리 | 반드시 "~입니다", "~되었습니다"로 수정 |
| **접속 부사 남발** | "또한", "그리고", "한편" 연속 사용 | 연속 2문장 이상 접속 부사로 시작 금지. 문장 자체의 논리 흐름으로 연결. |
| **과장 형용사** | "매우", "정말", "엄청나게" 삽입 | 수치로 대체하거나 삭제 |
| **번역체** | "~라고 할 수 있겠습니다", "~라는 점에서 주목할 만합니다", "~에 대해 논의하고자 합니다" | 직접적 표현으로 교체. 소리 내어 읽었을 때 어색하면 반드시 고침. |
| **주술 분리** | 주어와 술어 사이에 수식어가 3개 이상 끼어 의미 파악이 어려움 | 문장을 나누어 주어 바로 뒤에 술어가 오도록 재구성 |
| **한 문장에 정보 과다** | 하나의 문장에 여러 사실을 나열하여 이해가 어려움 | 한 문장에 하나의 정보만 담도록 분리 |
| **모호한 실행 지침** | "AI를 활용해 보세요", "자동화를 고려해 보세요" | 구체적 도구명 + 프롬프트 예시 + 소요 시간 명시 |
| **구조 스킵** | [3.5] 비즈니스 비용이나 [C] 리더의 결정 포인트를 빠뜨림 | 11개 섹션 체크리스트로 확인. 생략 시 "[해당 없음]" 명시. |
| **출처 누락** | 수치를 제시하면서 출처를 안 붙임 | 모든 수치에 (출처: 기관명, 연도) 형식 필수 |
| **단어 중간 줄바꿈** | HTML에서 word-break 미적용 | CSS에 `word-break: keep-all` 확인 |
| **히어로 이미지 lazy** | 썸네일 히어로에 `loading="lazy"` 적용 | 히어로는 LCP이므로 lazy loading 제외 |

## 15. 자주 하는 실수 (일반)

| 실수 | 교정 |
|------|------|
