대화만으로 AI 에이전트를 훈련한다: OpenClaw-RL

arXiv (Yinjie Wang, Ling Yang 외) Yinjie Wang, Xuyang Chen, Xiaolong Jin, Mengdi Wang, Ling Yang OpenClaw-RL: Train Any Agent Simply by Talking ↗
#AI에이전트 #강화학습 #OpenClaw #RLHF
3줄 요약

AI 에이전트 훈련에는 전문가가 만든 라벨링 데이터가 필수였습니다. OpenClaw-RL은 사용자 대화, 터미널 출력, GUI 변화처럼 에이전트가 이미 받고 있는 신호만으로 학습하는 방법을 제시합니다.

"이 행동이 좋았는가"를 판단하는 Binary RL과 "어떻게 고쳐야 하는가"를 교정하는 OPD, 두 방식을 결합해 Qwen3-4B 기준 개인화 점수를 0.17에서 0.81로 끌어올렸습니다.

대화, 터미널, GUI, SWE, 도구 호출 등 5가지 에이전트 유형을 하나의 비동기 파이프라인에서 동시에 훈련할 수 있어, 에이전트를 쓰면 쓸수록 자동으로 성능이 개선되는 구조입니다.

Roasting
AI 에이전트 훈련에 전문가 라벨링이 필수라고?
사용자가 "아니, 그거 말고"라고 한 마디만 하면 에이전트가 스스로 배운다. 당신의 데이터 파이프라인 비용, 얼마나 더 쓸 건가?
이 글, 왜 읽어야 하나

AI 에이전트를 조직에 맞게 훈련하려면 전문가가 수백에서 수천 건의 학습 데이터를 직접 만들어야 합니다. 비용도 크고 시간도 오래 걸립니다. OpenClaw-RL은 이 전제를 뒤집습니다. 사용자가 "그거 말고 다른 걸로 해줘"라고 말하면, 그 한 마디가 곧 훈련 신호가 됩니다. 에이전트가 이미 받고 있는 피드백으로 학습하니, 별도 라벨링 없이도 에이전트가 점점 똑똑해지는 구조입니다.

왜 지금 중요한가

AI 에이전트 도입은 빨라지고 있지만, 조직 맞춤 훈련은 여전히 느립니다. 라벨링 데이터를 만드는 데만 수주에서 수개월이 걸리기 때문입니다. OpenClaw-RL이 중요한 이유는 이 병목을 구조적으로 없앨 수 있기 때문입니다. 에이전트가 실제 업무 환경에서 쓰이면서 동시에 학습하는 방식이라, 도입과 훈련이 분리되지 않고 하나의 흐름으로 이어집니다.

핵심 인사이트

이 논문의 핵심은 "에이전트가 이미 받고 있는 신호를 어떻게 학습에 쓸 것인가"입니다. 주목할 포인트는 4가지입니다.

첫째, 새로운 데이터를 만들 필요가 없다.

에이전트가 행동하면 환경은 반드시 반응합니다. 사용자의 답변, 터미널 출력, GUI 화면 변화, 도구 호출 결과가 그것입니다. OpenClaw-RL은 이 반응을 "다음 상태 신호(next-state signal)"로 정의하고, 여기서 두 가지를 뽑아냅니다. "이 행동이 좋았는가"라는 평가 신호와, "어떻게 바꿔야 하는가"라는 지시 신호입니다. 에이전트가 작동하는 것만으로 학습 데이터가 자동 생성되는 셈입니다.

둘째, "평가"와 "교정"을 분리해서 결합한다.

Binary RL은 PRM(Process Reward Model)이 에이전트의 각 행동을 +1(좋음), 0(중립), -1(나쁨)으로 채점합니다. 무엇이 좋고 나빴는지를 구분하는 역할입니다. 반면 OPD(On-Policy Distillation)는 다음 상태에서 힌트를 추출해, 잘못된 행동을 토큰 수준에서 직접 교정합니다. 이 둘은 학습 궤적 자체가 다릅니다. Qwen3-4B + GSM8K 실험에서 Binary RL은 8단계 0.25 → 16단계 0.23으로 오히려 소폭 하락했고, OPD는 같은 구간에서 0.25 → 0.72로 급등했습니다. 하지만 두 방식을 결합하면 0.81로 가장 높은 성능을 기록합니다. 평가만으로는 방향을 잡기 어렵고, 교정만으로는 기준이 흔들리기 때문에 둘을 함께 써야 효과가 극대화됩니다.

셋째, "마지막 결과"가 아니라 "매 단계"를 평가해야 한다.

긴 작업에서 마지막 성공 여부만 보면, 중간에 어떤 행동이 기여했는지 알 수 없습니다. OpenClaw-RL은 PRM으로 매 단계마다 보상을 부여해 이 문제를 해결합니다. 효과 차이는 단계가 많은 작업에서 극명하게 드러납니다. 도구 호출 에이전트(Qwen3-4B, AIME 2024)에서 과정별 보상(0.30)은 최종 결과 보상(0.17)보다 76% 높았습니다. GUI 에이전트(Qwen3VL-8B, OSWorld)에서는 격차가 줄어들지만(0.33 vs 0.31), 여전히 과정별 보상이 우위를 보입니다.

넷째, 에이전트 유형이 달라도 하나의 파이프라인으로 훈련한다.

보통 에이전트 유형이 다르면 훈련 파이프라인도 따로 만들어야 합니다. OpenClaw-RL은 대화(Qwen3-4B), 터미널(Qwen3-8B), GUI(Qwen3VL-8B), SWE(Qwen3-32B), 도구 호출(Qwen3-4B) 5가지를 하나의 비동기 아키텍처에서 동시에 처리합니다. 핵심은 4개 모듈의 완전한 분리입니다. 모델 서빙(SGLang), 환경 호스팅(HTTP/API), 보상 판정(PRM), 정책 훈련(Megatron)이 각각 독립적으로 작동해 모듈 간 조정 오버헤드가 0입니다. 덕분에 터미널 128개, GUI/SWE 64개, 도구 호출 32개 환경을 동시에 돌릴 수 있습니다.

비즈니스 비용

에이전트 도입의 최대 병목은 훈련 데이터를 만드는 비용입니다. 조직 맞춤 에이전트를 훈련하려면 전문가가 직접 학습 데이터를 작성해야 합니다. 라벨링 비용, 품질 검수 인력, 도메인 전문가 참여까지 합산하면 에이전트 1개에 수천만 원에서 수억 원이 들고, 에이전트 수가 늘 때마다 이 비용은 선형으로 증가합니다.

이 비용 구조가 바뀔 수 있습니다. OpenClaw-RL 방식에서는 사용자가 에이전트를 쓰는 행위 자체가 훈련 데이터를 만들어냅니다. 논문에 따르면, 개인 에이전트는 36회의 상호작용만으로 유의미한 성능 향상을 보였습니다. 이 방식이 실 서비스에서 검증되면, 에이전트 확장의 한계 비용이 크게 낮아지고 도입 속도와 커스터마이징 범위를 동시에 넓힐 수 있습니다.

내일 당장 할 수 있는 것
1
현재 AI 에이전트가 받는 피드백 신호를 목록화하세요.
사용자 응답, 작업 성공/실패 로그, 재시도 패턴, 수정 요청 등 에이전트가 이미 받고 있는 신호를 정리합니다. 이 신호들이 OpenClaw-RL의 "다음 상태 신호"로 전환 가능한지 평가합니다. (2시간)
2
OpenClaw-RL GitHub 레포를 확인하세요.
github.com/Gen-Verse/OpenClaw-RL에서 코드 구조와 지원 환경(CUDA 12.9, Python 3.12, 기본 8GPU 구성)을 확인합니다. 자사 에이전트 아키텍처와 호환되는지 ML 엔지니어와 함께 검토합니다. (반나절)
3
에이전트 훈련 비용에서 라벨링이 차지하는 비중을 산출하세요.
현재 훈련 파이프라인에서 라벨링 데이터 구축에 들어가는 비용(인건비, 시간, 외주비)을 분리합니다. 이 비용이 전체 훈련 비용의 몇 %인지 파악하면, 라벨링 없는 학습이 가져올 비즈니스 임팩트를 정량화할 수 있습니다. (1시간)
리스크 & 주의점

프라이버시. 사용자 대화로 학습한다는 것은 대화 내용이 모델에 직접 반영된다는 뜻입니다. 개인정보나 기밀이 포함된 대화를 훈련에 쓰려면, 명시적 동의 절차와 데이터 처리 정책이 선행되어야 합니다.

피드백 품질. 사용자의 교정이 항상 정확하지는 않습니다. 잘못된 피드백이 쌓이면 성능이 오히려 악화됩니다. 논문에서는 PRM 다수결 투표(GUI의 경우 3회 독립 판정)로 노이즈를 걸러내지만, 실 서비스 환경에서도 이 필터가 충분한지는 아직 검증되지 않았습니다.

실 서비스 격차. 실험은 GSM8K 시뮬레이션, OSWorld, SWE-Bench 등 통제된 벤치마크에서 진행됐습니다. GPU 수, 훈련 시간, 연산 비용 같은 인프라 정보도 공개되지 않았습니다. 수천 명이 동시에 쓰는 환경에서의 안정성과 비용 효율성은 별도로 검증해야 합니다.

한 줄 정리

에이전트 훈련의 병목은 데이터가 아니라 데이터를 만드는 비용이었습니다. OpenClaw-RL은 사용자가 에이전트를 쓰는 행위 자체를 학습 데이터로 전환합니다. 36회 상호작용만으로 개인화 점수가 0.17에서 0.81로 뛴 결과는, "쓸수록 똑똑해지는 에이전트"가 실현 가능한 구조임을 보여줍니다.

리더의 결정 포인트
창업자 현재 에이전트 훈련 비용에서 라벨링이 차지하는 비중을 파악하고, 자동 학습으로 전환할 시점을 검토하라
팀장 에이전트가 받는 사용자 피드백 신호(대화 응답, 재시도 패턴, 수정 요청)를 로깅하고 있는지 확인하라. 로깅이 없으면 학습 전환 자체가 불가능하다
임원 사용자 데이터를 에이전트 학습에 활용하려면, 프라이버시 정책과 명시적 동의 절차를 선제적으로 수립하라
